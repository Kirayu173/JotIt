# v0.1 Stage Design

## Stage Goal

`v0.1` 的目标不是做出完整财务系统，而是做出一个你愿意每天打开使用的最小闭环：

- 打开 App 先看到自己的最近记录
- 点一下按钮，从底部拉起大面积记账弹窗
- 几步内完成一笔人民币收支录入
- 能快速回看、修改、删除并撤销
- 到月末自动产出规则型总结
- 也能手动查看任意月份和任意区间的回顾结果

这个阶段默认：

- Android 优先
- 离线可用
- 数据只存本地
- 不依赖账号
- 不接 AI / OCR

## Confirmed Decisions

### 已确认项

1. 平台优先级：`Android 优先`
2. 一级导航：`记账 / 回顾 / 设置`
3. 记账页形态：默认进入记账页，主视图为列表；点击主按钮后，从底部拉起覆盖大半屏幕的记账弹窗
4. 回顾页范围：`月度概览 + 自动总结 + 流水列表` 同页承载
5. 分类策略：`内置默认分类 + 支持新增自定义分类 + 不做二级分类`
6. 时间策略：默认当前时间，支持手动修改日期与时间
7. 删除策略：`软删除 + 短时撤销`
8. 总结策略：支持手动查看任意月份与任意区间，粒度覆盖 `周 / 月 / 季 / 年 / 自定义区间`，并在月末自动生成当月总结
9. 列表策略：记账页使用列表展示记录，支持向上滑动加载更早记录，但只加载到某个时间门限
10. 视觉风格：`浅色优先，简洁明快`
11. 币种范围：`v0.1 仅支持人民币 CNY`
12. 记账页列表最早加载时间门限：`365 天`
13. 记账页列表分组方式：`按天分组`
14. 记账弹窗金额输入方式：`系统数字键盘`
15. 默认分类集合粒度：`8 个支出 + 3 个收入`
16. 回顾页默认时间粒度：`月`

## Finalized Detail Decisions

下面这些细项已经确认，将直接进入 `tasks.md` 和后续实现。

### D1 - 记账页列表向上加载的最早时间门限

- 推荐：`365 天`
- 已确认：`365 天`
- 备选：
  - `180 天`
  - `730 天`
  - 不设门限，全部可滚动加载
- 推荐原因：
  - 365 天足以覆盖个人年度消费回顾
  - 性能和体验更稳，避免默认列表无限增长
  - 更早的历史仍然可以通过回顾页按区间查询，不需要全塞在默认列表里

### D2 - 记账页列表分组方式

- 推荐：`按天分组`
- 已确认：`按天分组`
- 备选：
  - 平铺单列表
  - 按月分组
- 推荐原因：
  - 日常记账时，人更容易按“今天 / 昨天 / 某日”定位
  - 比按月分组更利于快速修正刚刚录错的记录
  - 比纯平铺更易读

### D3 - 记账弹窗金额输入方式

- 推荐：`系统数字键盘`
- 已确认：`系统数字键盘`
- 备选：
  - 自定义计算器键盘
  - 仅普通文本输入
- 推荐原因：
  - `v0.1` 优先控制实现复杂度
  - 系统数字键盘已足够快，也更稳
  - 后续若你真的高频使用后觉得不够快，再升级到自定义键盘

### D4 - 默认分类集合粒度

- 推荐：`8 个支出 + 3 个收入`
- 已确认：`8 个支出 + 3 个收入`
- 备选：
  - `5 个支出 + 2 个收入`
  - `12+ 个支出 + 5+ 个收入`
- 推荐原因：
  - 太少会很快不够用
  - 太多会增加选择成本
  - 8+3 对个人记账来说通常是很好的起点

推荐的默认分类：

- 支出：餐饮、交通、日用、购物、娱乐、居住、医疗、其他
- 收入：工资、报销、其他

### D5 - 回顾页默认打开的时间粒度

- 推荐：`月`
- 已确认：`月`
- 备选：
  - `周`
  - `最近 30 天`
  - `季度`
- 推荐原因：
  - 月度是最符合记账习惯的回顾单位
  - 也最适合承接“月末自动总结”
  - 周、季、年和自定义区间可作为切换选项

## Information Architecture

### 一级导航

- `记账`
- `回顾`
- `设置`

### 设计意图

- `记账` 是高频入口，负责“看最近记录 + 快速新增”
- `回顾` 是解释页面，负责“看时间范围总结 + 看流水 + 改记录”
- `设置` 只承载低频配置，不进入高频主链路

## Screen Design

### 1. 记账页

### 页面职责

- 展示最近记录列表
- 提供固定可见的新增入口
- 在不离开列表上下文的前提下完成新增
- 让“录完一笔 -> 看见它 -> 立刻改”成为自然动作

### 页面结构

1. 顶部导航栏
   - 页面标题：`记账`
   - 右侧可放简短时间范围提示，例如“近 30 天已支出”

2. 轻量概览区
   - 今日支出
   - 本月支出
   - 本月结余

3. 记录列表
   - 默认按时间倒序
   - 支持上拉加载更早记录
   - 支持点击进入编辑

4. 主操作按钮
   - 固定悬浮在右下角
   - 点击后拉起底部大弹窗

### 列表设计建议

- 每条记录展示：
  - 分类图标
  - 分类名
  - 备注或商家
  - 金额
  - 时间
- 支出金额用偏暖色，收入金额用偏绿色
- 记录之间用轻分割和留白区分，不做厚重卡片

### 交互设计

- 进入页面后不直接弹出记账层，先显示列表，减少压迫感
- 点击主按钮后，弹窗从底部上滑，覆盖大约 `65% - 75%` 屏幕
- 保存后弹窗关闭，并把新记录插入列表顶部
- 删除后显示短时撤销条

## 2. 记账弹窗

### 设计目标

- 快
- 不跳页
- 可连续录入
- 不要求填写过多字段

### 推荐布局

1. 顶部栏
   - 标题：`记一笔`
   - 关闭按钮

2. 收支切换
   - `支出 / 收入`

3. 金额输入区
   - 大字号金额
   - 使用系统数字键盘

4. 分类快捷区
   - 默认分类网格或横向快捷组
   - 可进入“更多分类”

5. 附加信息区
   - 备注
   - 日期时间

6. 底部操作区
   - 保存
   - 保存并继续

### 为什么不用独立页面

- 你要的是快记账，不是沉浸式表单
- 保留底层列表上下文，录完能立即感知结果
- 比全屏跳转更符合工具型产品的节奏

## 3. 回顾页

### 页面职责

- 展示总结
- 展示趋势
- 展示同一时间范围下的流水
- 允许从回顾结论直接跳去修正记录

### 页面结构

1. 顶部时间粒度切换
   - 周
   - 月
   - 季
   - 年
   - 自定义

2. 时间范围选择器
   - 当前月 / 当前季度 / 某个区间

3. 概览卡片
   - 总收入
   - 总支出
   - 净结余
   - 主要支出分类

4. 自动总结卡片
   - 2-4 条规则型结论
   - 例如“本月餐饮支出高于上月 18%”

5. 趋势或分类分布模块
   - `v0.1` 只做最基础一种图表即可
   - 推荐优先做分类占比或时间趋势二选一

6. 同时间范围的流水列表
   - 支持筛选
   - 支持点进编辑

### 设计重点

- 回顾页不是“第二个首页”
- 它是一个解释层，把“这段时间花了什么钱”讲清楚
- 结论和明细必须同页，避免来回跳转

## 4. 设置页

`v0.1` 保持克制，只放：

- 分类管理
- 主题/视觉偏好预留
- 数据说明
- 关于页

不放：

- 云同步
- AI 设置
- OCR 设置
- 复杂账号能力

## Summary Design

### `v0.1` 自动总结范围

支持：

- 手动查看任意月份总结
- 手动查看任意区间总结
- 粒度切换：周 / 月 / 季 / 年 / 自定义区间
- 到月末自动生成当月总结快照

### `v0.1` 规则型总结最小集合

1. 本期总收入
2. 本期总支出
3. 净结余
4. 最大支出分类
5. 与上一个可比周期的变化
6. 高频分类提示

### 文案风格

- 简短
- 直接
- 不做理财建议
- 不做“教你花钱”的语气

示例：

- “本月支出 3,862 元，较上月增加 12%。”
- “餐饮是本月最大支出类别，占总支出的 28%。”
- “本月收入高于支出 1,240 元，整体保持结余。”

## Visual Direction

### 风格关键词

- 浅色
- 明快
- 安静
- 工具感
- 留白清晰

### 推荐视觉方向

- 背景：偏暖白或浅灰白
- 主色：偏青绿或墨绿色
- 支出强调：偏珊瑚红或暖橙红
- 收入强调：偏绿色
- 文字：深灰而不是纯黑

### 不建议

- 大面积深色背景
- 拟物强阴影
- 理财 App 式高饱和渐变
- 过多装饰插画

## Data and Performance Notes

### 本地优先

- 使用 SQLite 作为唯一数据源
- 记账和回顾核心流程均不依赖网络

### 列表加载

- 记账页只加载一个受限时间窗内的数据
- 更大范围回顾通过回顾页查询完成

### 删除与撤销

- 删除采用软删除
- 撤销条时效建议控制在数秒内

## Risks

### 风险 1 - 记账页既要显示列表又要保持“快录”

缓解方式：

- 列表做轻，不做复杂卡片
- 悬浮按钮足够醒目
- 顶部概览只保留 2-3 个数字

### 风险 2 - 回顾页很容易做成第二个统计中心

缓解方式：

- `v0.1` 只保留一个图表模块
- 自动总结优先，复杂报表后置

### 风险 3 - 区间粒度太多会拉高实现复杂度

缓解方式：

- 先统一做时间范围查询和聚合接口
- UI 只在上层提供几种 preset
